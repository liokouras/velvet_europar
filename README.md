# Artifact Overview of Velvet: Parallel Divide-and-Conquer in Safe Rust

This artifact contains all relevant resources to reproduce the results presented in the paper *Velvet: Parallel Divide-and-Conquer in Safe Rust*. There are two key parts: the first reproduces the tables and figures in the paper using our original data; the second runs a reduced version of all experiments and produces corresponding tables and figures from the output. All source code and data is hosted in this repository.

A fully formatted version of this document is available as [as a pdf](OverviewDocument.pdf).

## 1. Getting Started Guide

### 1.1 Hardware Requirements

The artifact should run on any multi-core x86-64 Linux workstation or cluster node. The original experiments used a dual-socket AMD EPYC 7702 with 128 physical cores. However, this artifact is adjusted for a reduced evaluation, targeting a maximum of 64 cores; more cores are not required.

### 1.2 Operating System & Software Dependencies

To compile and run the experiments, the following software components are needed. The listed versions do not have to match exactly, as the experiments should run with any modern version of each component.

| Component | Version used in original setup |
|---|---|
| Linux OS | Rocky Linux 9.7 (Blue Onyx) |
| Rust toolchain | rustc 1.93.1 (stable), Cargo (via rustup) |
| C compiler | Clang 20.1.8 |
| Make | GNU Make 4.3 |
| OpenMP | 5.1 (via Clang) |
| OpenCilk | 2.1 (built with Clang, Git 2.47.3 and CMake 3.26.5) |

To process data and produce the tables and figures in the paper, Python 3.7 and the packages `matplotlib`, `numpy`, and `pandas` are required.

### 1.3 Installation

- **Python dependencies** — install using pip:
  ```bash
  pip3 install numpy matplotlib pandas
  ```

- **Rust toolchain** — follow [these instructions](https://rust-lang.org/tools/install), or run:
  ```bash
  curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
  ```

- **Clang with OpenMP** — typically available through the system package manager, though tends to require root access. On RHEL/Rocky systems: `sudo dnf install clang openmp-devel`. On Debian/Ubuntu systems: `sudo apt install clang libomp-dev`.

- **OpenCilk** — follow the [quick start instructions](https://www.opencilk.org/doc/users-guide/build-opencilk-from-source/). After installation, set the `$OPENCILK_HOME` environment variable to point to the `build` folder created during installation:
  ```bash
  export OPENCILK_HOME=/path/to/build
  ```

- **The Artifact** — clone and enter the main directory:
  ```bash
  git clone https://github.com/liokouras/velvet_europar.git
  cd velvet_europar
  ```

#### Smoke Test

To verify the environment configuration, run:
```bash
bash smoke_test.sh
```

This test checks for each dependency independently and prints either a confirmation or a warning for each:

| Dependency | How it is checked |
|---|---|
| Python dependencies | invoked with `python3` |
| Rust toolchain | `rustc` and `cargo`, expected on PATH |
| Clang with OpenMP | `clang -fopenmp`, expected on PATH |
| OpenCilk | `$OPENCILK_HOME/bin/clang -fopencilk`; `$OPENCILK_HOME` must be set |
| `lscpu` | used to auto-detect physical core count across sockets |

A successful smoke test prints four `VERIFIED` lines. At minimum, only Python is needed for data processing and Rust to run the basic benchmarks. If OpenMP and/or OpenCilk cannot be configured, the experiments will still run but skip the corresponding C versions; this affects Figure 2, which will be missing the corresponding data points.

### 1.4 Data Dependencies

The raw output data from our own experiments is included in the artifact. Most input data for the experiments is generated by the benchmarks themselves, except for the Barnes-Hut benchmark. This input is too large to host directly in the repository and is part of the GitHub release assets. It is downloaded automatically by the relevant run script; no manual action is required. The download should only not take more than a minute.

## 2. Step-by-Step Instructions to Reproduce Results

The artifact has the following high-level structure:

```
velvet_europar/
├── benchmarks/
│   ├── c/
│   ├── runscripts/
│   └── rust/
├── data_processing/
│   ├── data/
│   └── scripts/
├── process_data.sh
├── run_exp.sh
└── smoke_test.sh
```

### 2.1 Generating Tables and Figures from Raw Data

> *This part only requires the Python dependencies listed above and should complete within a minute.*

Run the following command from inside `velvet_europar`:
```bash
bash process_data.sh
```

This runs the data-processing pipeline using our original raw data and produces:
- The last four columns of **Table 1** (page 7)
- **Table 2** (page 8)
- **Figure 1** (page 9)
- **Figure 2** (page 10)

Tables are printed to `data_processing/processing_output.txt` and figures are saved in `data_processing/figs/`.

### 2.2 Running the Benchmarks

> *This part requires a multi-core Linux machine and, at minimum, the Rust toolchain. OpenMP and OpenCilk are necessary to reproduce the results in Figure 2.*

Run the following command from inside `velvet_europar`:
```bash
bash run_exp.sh [--cores=N]
```

This runs a **reduced version** (see below) of all experiments, then calls the data-processing pipeline on the generated data, producing the same outputs as Section 2.1. If the C dependencies are present, Figure 2 is also produced.

#### The `--cores=N` option

The optional `--cores=N` argument overrides the auto-detected physical core count. By default, the script detects this automatically via `lscpu`, but if this is unavailable, `--cores=N` **must** be provided. In the reduced configuration, the core count is capped at 64 regardless of the value detected or provided. For example, to run on 32 cores:
```bash
bash run_exp.sh --cores=32
```

#### Reduced configuration

The default setup is reduced compared to the paper in the following ways:

| | Reduced (this artifact) | Full (paper) |
|---|---|---|
| Thread counts | capped at 64 | up to 128 |
| Runs per data point | 3 (first discarded) | 6 (first discarded) |
| Total runtime (our hardware) | ~4.5 hours | ~48 hours |

Problem sizes per benchmark (compare with Table 1, column 2 in the paper):

| Benchmark | `fib` | `adapint` | `tsp` | `nqueen` | `bh` | `matmul` | `sort` |
|---|---|---|---|---|---|---|---|
| Problem size | n=48 | [0, 8e5], eps=1e-4 | 18 cities | N=15 | 1e6 bodies | 4096×4096 | 1e9 elements |

Serial thresholds remain unchanged.

Additionally, the **Rayon, Demo-Strassen** variant of `matmul` is omitted as it takes too long to execute. This affects one line in the `matmul` plot of Figure 1.

#### Full configuration

The full original experiments can be run with:
```bash
bash run_exp.sh full
```
Note that This will likely take **days** to complete.

### 2.3 Interpreting the Results

**Section 2.1** output should match the paper's tables and figures exactly (same raw data), with one exception: Table 2 is transposed in the paper for better layout and spacing.

**Section 2.2** output will naturally differ in absolute values due to the reduced configuration and differing hardware. The following trends should be reproduced:

- **Table 1** basic benchmark information; no specific trends expected.
- **Table 2** work overhead factors close to 1.00, *except* for `fib` with no threshold, which will have a large overhead.
- **Figure 1** near-linear speedup for the top-row benchmarks; sub-linear scaling for the bottom row. Velvet and Rayon perform comparably. There will be no line for **Rayon, Demo-Strassen**.
- **Figure 2** similar trends as Figure 1, with Velvet performing comparably with the C frameworks. If any C dependencies are missing, the corresponding data will be absent.
